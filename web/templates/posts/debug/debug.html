Debugging: : The Power of Reverse Engineering

In general,

Debuggers are designed to allow developers to measure and control the internal state and execution of a program.

Assembly-level debuggers

  • Kernel vs. User-Mode Debugging

    User-Mode

    the debugger is running on the same system as the code being debugged.

    Kernel-Mode

    Kernel debugging is performed on two systems because there is only one kernel. One system runs the code that is being debugged, and another runs the debugger.

    Breakpoints ⇒ pause execution and allow you to examine a program’s state, it take much longer to run than ordinary instructions

    Types of breakpoints :

    • Software Execution ⇒ stop when a particular instruction is executed.
      • How the debugger implements a software breakpoint ?

        by overwriting the first byte of an instruction with 0xCC, the instruction for INT 3, the breakpoint interrupt designed for use with debuggers. When the 0xCC instruction is executed, the OS generates an exception and transfers control to the debugger.

    • Hardware Execution ⇒ implemented within the processor's architecture and provide a powerful way to halt program execution when a certain condition is met.
      • Advantage
        • it can be set to a specific address in memory.
        • it can be set to break on access rather than on execution ( ex: when a memory location read or write ).
      • Drawback
        • only four hardware registers store breakpoint addresses. ( DR0 through DR3 )
        • The debug control register (DR7) stores information on whether the values in DR0 through DR3
        • Malicious programs can modify these registers.
      • in x86 chips, by setting the General Detect flag in the DR7 register, it will detects mov instructions that access the debug registers, by setting a breakpoint.
    • Conditional ⇒ implemented as software breakpoints, break only if a certain condition is true.

    Exception ⇒ the principal way that a debugger gains control of a running program

    First- and Second-Chance Exceptions

    When an exception occurs while a debugger is attached, the program being debugged stops executing, and the debugger is given a first chance at control. If an application does not handle the exception, the debugger is given another chance to handle it—the second-chance exception.

    Common Exceptions :

    • INT 3 exceptions
    • Single-stepping ⇒ the trap flag (FLAGS reg) is used for single-stepping. when it is set, the processor executes one instruction and then generates an exception.
    • memory-access violation exception ⇒ code tries to access a location that it cannot access
    • Certain instructions can be executed only when the processor is in privileged mode. When the program attempts to execute them outside privileged mode, the processor generates an exception.
